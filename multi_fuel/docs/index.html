Aviary Extension

multi_fuel

Use a different CSV engine deck for each mission phase and retrieve per-engine total fuel burn as an OpenMDAO output variable.

Overview

Aviary's standard engine model uses a single CSV performance deck across all flight phases. multi_fuel extends this with MultiEngineTableBuilder, which dispatches a different EngineDeck for each named phase (climb, cruise, descent, …). After the mission, per-engine fuel burn is summed and exposed as Mission.TOTAL_FUEL_MULTI — an array whose length equals the number of unique engine CSV files in the mapping.

Additive only. All changes to Aviary are additive. Existing engine models and mission setups are not affected.
climb ──> turbofan_28k.csv ──┐
cruise ──> turbofan_22k.csv ──┤ sum by unique CSV
descent ──> turbofan_22k.csv ──┘

Mission.TOTAL_FUEL_MULTI = [fuel_28k, fuel_22k]

Contents

  • table_builder.py EngineTableBuilder, MultiEngineTableBuilder, MultiEngineFuelBurnComp

EngineTableBuilder

A thin wrapper around Aviary's EngineDeck that accepts a CSV path directly at construction time. Useful on its own or as a building block inside MultiEngineTableBuilder.

python
from multi_fuel.table_builder import EngineTableBuilder

engine = EngineTableBuilder(
    name='my_engine',
    csv_path='models/engines/turbofan_22k.csv',
)

Accepts both Aviary-relative paths (resolved via get_path) and absolute paths.

MultiEngineTableBuilder

Selects a different EngineTableBuilder CSV deck for each named mission phase. All CSV files are pre-loaded at construction time. During the mission, the correct deck is dispatched based on the phase name injected into subsystem_options by configure_phase_info.

Constructor

signature
MultiEngineTableBuilder(
    name: str = 'multi_engine_table',
    phase_engine_map: dict = None,
)
ParameterTypeDescription
name str Subsystem label. Must be unique among all external subsystems passed to load_external_subsystems.
phase_engine_map dict Mapping of phase name → CSV path. Multiple phases may share the same CSV; those phases are summed together in Mission.TOTAL_FUEL_MULTI. Index order in the output array follows first appearance of each unique CSV.

Usage

python
from copy import deepcopy
from aviary.core.aviary_problem import AviaryProblem
from aviary.models.missions.energy_state_default import phase_info
from multi_fuel.table_builder import MultiEngineTableBuilder

engine = MultiEngineTableBuilder(
    name='multi_engine_table',
    phase_engine_map={
        'climb':   'models/engines/turbofan_28k.csv',
        'cruise':  'models/engines/turbofan_22k.csv',
        'descent': 'models/engines/turbofan_22k.csv',
    },
)

# Inject phase names before passing phase_info to AviaryProblem.
phase_info = engine.configure_phase_info(deepcopy(phase_info))

prob = AviaryProblem()
prob.load_inputs(aviary_inputs, phase_info)
prob.load_external_subsystems([engine])
prob.check_and_preprocess_inputs()
prob.build_model()
# ... add_driver, add_design_variables, add_objective, setup, run ...

prob.run_aviary_problem()

from aviary.variable_info.variables import Mission
print(prob.get_val(Mission.TOTAL_FUEL_MULTI, units='lbm'))

configure_phase_info

Injects the current phase name into each flight phase's subsystem_options so build_mission can dispatch to the correct engine deck. Modifies phase_info in-place and returns it. Skips pre_mission and post_mission entries.

Call order matters. configure_phase_info must be called before AviaryProblem.load_inputs.

Output variable: Mission.TOTAL_FUEL_MULTI

After the mission runs, per-engine fuel burn is available via prob.get_val(Mission.TOTAL_FUEL_MULTI, units='lbm'). Fuel burn for each phase is the change in gross mass from the start to the end of that phase (mass[t=0] − mass[t=final]), then summed across phases that share the same engine CSV.

IndexCSV (example)Phases contributing
0 turbofan_28k.csv climb
1 turbofan_22k.csv cruise + descent
The array length equals the number of unique CSV paths in phase_engine_map, in order of first appearance. Mission.TOTAL_FUEL_MULTI tracks only trajectory phases; taxi and takeoff fuel (if modelled separately) are not included.

Changes made to Aviary

Four Aviary source files were modified. All changes are additive; existing engine models and mission setups are unaffected.

aviary/variable_info/variables.py

additive

Added TOTAL_FUEL_MULTI = 'mission:total_fuel_multi' to the Mission class alongside the existing TOTAL_FUEL entry.

aviary/variable_info/variable_meta_data.py

additive

Added an add_meta_data entry for Mission.TOTAL_FUEL_MULTI:

FieldValue
unitslbm
default_value0.0
desc Total fuel burned per unique engine table, ordered by first appearance in phase_engine_map. Array length equals the number of unique engine CSVs.

aviary/subsystems/propulsion/engine_model.py

extended

Two new methods added to EngineModel, both returning empty defaults so all existing engine models are unaffected:

MethodDefault returnPurpose
get_post_mission_promotes_outputs() [] Returns the promotes_outputs list used when adding the engine's post-mission component to PostMissionGroup. Not needed when PostMissionGroup.configure promotion handles the variable automatically (as it does here), but available for future subclasses.
get_traj_connections(regular_phases) [] Returns a list of (src, tgt, src_indices) tuples. AviaryGroup.link_phases iterates these and calls self.connect for each, wiring trajectory mass timeseries into the post-mission component's inputs.

aviary/core/aviary_group.py

extended

Two targeted additions inside AviaryGroup:

add_post_mission_systems

After the existing loop over self.subsystems, a new loop iterates self.engine_models and calls build_post_mission on each. If a component is returned it is added to PostMissionGroup without explicit promotes. PostMissionGroup.configure then calls promote_aircraft_and_mission_vars, which automatically promotes any mission:* output — including Mission.TOTAL_FUEL_MULTI — up to the top level.

link_phases

After the standard trajectory-to-state_output connections, a new loop calls get_traj_connections(self.regular_phases) on each engine model and registers the returned (src, tgt, src_indices) connections. This is how traj.{phase}.timeseries.mass[0] and traj.{phase}.timeseries.mass[-1] are wired into MultiEngineFuelBurnComp's inputs for each phase.
